The problem

Voice agents generate responses from raw transcripts. Then they hallucinate context.

The architecture most voice AI systems ship with is: microphone → speech-to-text → LLM → response. Every step in that chain loses something the next step needed. Here's where it breaks down.

Failure mode 1

No temporal context

A batch transcript gives you the whole conversation after it finishes. During the conversation, the agent has no idea where in the audio timeline it is. It can't reason about timing, pacing, or what was just said vs. said three minutes ago.

Failure mode 2

No speaker grounding

The transcript doesn't know who said what. The agent has to guess from context — which means it sometimes assigns user statements to the assistant and vice versa. A grounded agent knows the speaker identity for every segment it reasons over.

Failure mode 3

No session state accumulation

Partial transcripts arrive as text with no state tracking. The agent doesn't accumulate a coherent picture of the conversation — it processes text fragments it has to re-interpret every time, without the benefit of a growing context window it's actually reasoning over correctly.

Failure mode 4

The hallucination amplifier

Without timestamps, the agent can't say "at 4.3 seconds, the user said X." It has to say "the user said X" — and that statement has no verification path. When the agent confabulates context, there's no artifact to check it against.

Failure mode 5

Late context = wrong responses

In a real-time voice call, you can't wait for a batch transcript to finish before you respond. But early responses are made with less context. A streaming architecture gives you partial context in real time — enough to respond accurately without waiting for the full audio.

Failure mode 6

No citation path for downstream systems

If your voice agent handles a support call, your QA team can't verify what the agent said was grounded in what the user actually said. Without timestamps, every agent response is post-hoc — you can't replay the evidence that justified it.

Why timestamps fix it

A timestamped segment is a citable artifact.

When every transcript fragment has a temporal address, the agent can say "at 4.3 seconds, the user said X" — and that statement can be verified. That's the difference between a statement with evidence and a hallucination waiting to happen.

Without timestamps

  • Agent: "Based on what you said about pricing..."
  • No way to verify which moment the agent referenced
  • No way to replay what "pricing" actually referred to
  • QA cannot verify agent response accuracy
  • Context window is text-only — no temporal dimension
  • Agent loses track of conversation state across turns

With timestamped streaming

  • Agent: "At 4.3s, you mentioned you're on the Pro plan..."
  • That statement maps to a specific audio segment
  • The segment is a verifiable artifact
  • QA can replay segment S2 (4.3s–8.1s) and confirm
  • Session state accumulates across timestamps
  • Agent reasons over temporal windows, not just text
The mechanism

WebSocket streaming: partial transcripts as they happen.

Batch transcription gives you the whole transcript after the fact. WebSocket streaming gives you timestamps at 300ms granularity — real-time context, not post-hoc reconstruction.

1

Audio chunk arrives from microphone

A stream of raw audio bytes — from a live call, a microphone input, or a streaming source. Each chunk is processed as it arrives, not accumulated and processed at end-of-stream.

2

Partial transcript emitted with timestamps

Signal Loom WebSocket returns a partial transcript object for each processed segment — with start_seconds, end_seconds, text, and language. The segment is available to the agent immediately, before the full audio finishes.

3

Session state accumulates across chunks

Each partial transcript appends to session state. The agent builds a temporal picture of the conversation as it happens — not reconstructs it after the fact. Context windows are populated with timestamped evidence, not raw text fragments.

4

Agent reasons over timestamps, not just text

With temporal context, the agent can weight recency, track what was said earlier vs. later, and cite specific moments when responding. Its responses are grounded in the actual audio timeline, not a text approximation of it.

WebSocket streaming partial response — real-time format
// WebSocket message: partial transcript { "type": "transcript", "text": "I want to upgrade my plan", "start_seconds": 4.3, "end_seconds": 8.1, "language": "en" } // Session state accumulates: session_state = [ { "segment_id": "S1", "start": 0.0, "end": 4.3, "text": "Hi, I'd like to talk about my account" }, { "segment_id": "S2", "start": 4.3, "end": 8.1, "text": "I want to upgrade my plan" }, // ... accumulating in real time ] // Agent response is grounded in timestamped session state: "Based on your request at 4.3 seconds — you're currently on the Starter plan..."

The key property: Each partial transcript has a start_seconds and end_seconds. The agent can reference specific moments. When it says "at 4.3 seconds," that maps to an actual audio artifact — not a text approximation of what was probably said.
